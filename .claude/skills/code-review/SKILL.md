# Codex Code Review

`codex review` コマンドを使い、作業ブランチの変更に対してコードレビューを実行する。

> **前提条件**: `codex` が利用可能であること。

---

## 起動条件

### パターン A: ユーザーリクエスト

以下のようなフレーズで起動する:

- 「レビューして」「コードレビュー」「コードを確認して」「差分を見て」
- 「review」「code review」

### パターン B: セルフレビュー

Claude Code がコードを変更した後、品質チェックのために自発的に実行する。以下の場面で実行する:

- 複数ファイルにまたがる変更を行った後
- リファクタリングを実施した後
- ユーザーの指示が曖昧で意図通りの実装か確信がないとき

---

## 実行手順

### ステップ 1: 前提確認

```bash
which codex
```

見つからない場合はユーザーに codex CLI のインストールが必要であることを伝えて終了する。

### ステップ 2: ベースブランチの特定

```bash
git rev-parse --verify main 2>/dev/null && echo "BASE=main" || \
  (git rev-parse --verify master 2>/dev/null && echo "BASE=master")
```

どちらも存在しない場合は `--base` なしで `--uncommitted` のみ使用する。

### ステップ 3: 変更内容の把握

```bash
# ベースブランチとの差分ファイル一覧
git diff --name-only <BASE>...HEAD

# 未コミット変更（ステージ済み + 未ステージ）
git diff --name-only HEAD

# 未追跡ファイル
git ls-files --others --exclude-standard

# ブランチ上のコミットメッセージ（変更意図の把握）
git log --oneline <BASE>..HEAD
```

変更ファイルが 0 件かつ未コミット変更もない場合は「レビュー対象の変更がありません」と伝えて終了する。

### ステップ 4: 関連ファイルの特定

変更ファイル一覧をもとに、以下の関連ファイルを特定する:

1. **テストファイル**: 変更されたソースに対応するテストファイル（`*_test.*`, `*.spec.*`, `test_*.*`, `__tests__/` 配下）
2. **設定ファイル**: 変更がインポートパスや依存関係に影響する場合（`package.json`, `tsconfig.json`, `Makefile`, `Dockerfile` 等）
3. **インポート元**: 変更されたモジュールを import/require しているファイル

```bash
# 例: 変更ファイルをインポートしているファイルを探す
git grep -l "import.*<変更ファイル名>" -- '*.ts' '*.tsx' '*.js' '*.jsx'
git grep -l "from ['\"].*<変更モジュール名>['\"]" -- '*.py'
```

### ステップ 5: codex review の実行

> **制約**: `--base`、`--uncommitted`、`--commit`、`[PROMPT]` は全て排他。同時指定不可。
> カスタム指示は `--title` 経由でのみ付与できる。
>
> | コマンド形式 | 対象 |
> |---|---|
> | `codex review --base <BRANCH> --title "..."` | ベースブランチとのコミット済み差分 |
> | `codex review --uncommitted --title "..."` | ステージ済み + 未ステージ + 未追跡 |
> | `codex review "prompt"` | プロンプトのみ（差分指定なし） |

#### パターン A: ユーザーリクエスト

コミット済み変更と未コミット変更を2段階でレビューする。

**ステップ 5a: コミット済み変更のレビュー（ベースブランチとの差分がある場合）**

`--title` にブランチ名・変更概要・関連ファイル情報を含めてコンテキストを与える。

```bash
codex review \
  --base <BASE> \
  --title "<ブランチ名> | 変更概要: <概要> | 関連ファイル: <ステップ4の結果> | 観点: バグ,エッジケース,セキュリティ,パフォーマンス,可読性,テスト網羅性,変更漏れ"
```

**ステップ 5b: 未コミット変更のレビュー（未コミット変更がある場合）**

```bash
codex review \
  --uncommitted \
  --title "<ブランチ名> 未コミット変更 | 変更概要: <概要> | 関連ファイル: <ステップ4の結果> | 観点: バグ,エッジケース,セキュリティ,パフォーマンス,可読性,テスト網羅性,変更漏れ"
```

> **注意**: コミット済み変更がなく未コミット変更のみの場合はステップ 5b のみ実行する。未コミット変更がない場合はステップ 5a のみ実行する。

結果をそのままユーザーに提示する。重大な指摘があればハイライトし、修正提案を行う。

#### パターン B: セルフレビュー

未コミット変更にフォーカスする。コミット済み変更もある場合は2段階で実行する。

**コミット済み変更がある場合、まずベースブランチとの差分をレビュー:**

```bash
codex review \
  --base <BASE> \
  --title "セルフレビュー: <タスク要約> | 変更ファイル: <一覧> | 関連ファイル: <ステップ4の結果>"
```

**未コミット変更のレビュー:**

```bash
codex review \
  --uncommitted \
  --title "セルフレビュー: <タスク要約> | タスク: <依頼内容> | 観点: 要件充足,副作用,コードスタイル,エラーハンドリング,テスト要否,変更漏れ"
```

### ステップ 6: 結果の処理

#### パターン A: ユーザーリクエスト

codex の出力をそのまま表示して終了。

#### パターン B: セルフレビュー（修正ループ）

**指摘が全て解消されるか、対応不要と判断するまでループを繰り返す。**

```
ループ:
  1. codex review の出力を解析し、指摘事項を抽出する
  2. 各指摘を以下に分類する:
     - 【修正する】: バグ、論理エラー、要件の未充足、セキュリティ問題 → 修正を実施
     - 【対応不要】: スコープ外、既存コードの規約に沿っている、トレードオフを考慮した上での設計判断 等
  3. 修正すべき指摘がある場合:
     a. 修正を実施する
     b. 再度 codex review を実行する（ループ先頭に戻る）
  4. 全指摘が解消 or 対応不要になったらループ終了
```

**ループ終了後、ユーザーに以下を報告する:**

- 修正した内容の一覧（何をどう直したか）
- 対応不要と判断した指摘の一覧と、それぞれの理由
  - 例:「○○ は既存コードベースの規約に従った記述のため対応不要」
  - 例:「○○ は今回のタスクスコープ外の変更のため対応不要」
  - 例:「○○ はパフォーマンスと可読性のトレードオフを考慮し現状維持」
- 最終レビュー結果（問題なし or 残課題あり）

---

## エラーハンドリング

| 状況 | 対応 |
|------|------|
| `codex` が見つからない | codex CLI のインストールが必要であることを伝えて終了 |
| ベースブランチが見つからない | `--uncommitted` のみでレビュー実行 |
| 変更が 0 件 | 「レビュー対象の変更がありません」と伝えて終了 |
| codex がエラーを返した | エラー出力を表示し、考えられる原因を伝える |
| codex がタイムアウト | 変更範囲が大きすぎる可能性を伝え、範囲を絞ったレビューを提案 |
