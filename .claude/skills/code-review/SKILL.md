# Multi-Agent Code Review

3 つのレビューエージェントが並列にレビューし、親 Claude がとりまとめる反復レビュープロセス。

## レビューエージェント

| エージェント | 種別 | 専門領域 |
|---|---|---|
| **Agent A — Logic Reviewer** | Claude サブエージェント | バグ・ロジックエラー・エッジケース・テスト網羅性・エラーハンドリング・データ整合性 |
| **Agent B — Security & Design Reviewer** | Claude サブエージェント | セキュリティ・認証/認可・マルチテナント分離・パフォーマンス・設計・命名規約・破壊的変更 |
| **Agent C — Codex Reviewer** | Claude サブエージェント (内部で codex CLI 実行) | 全観点の総合レビュー |

---

## 起動条件

### パターン A: ユーザーリクエスト

以下のようなフレーズで起動する:

- 「レビューして」「コードレビュー」「コードを確認して」「差分を見て」
- 「review」「code review」
- 「3ラウンドでレビューして」（ラウンド数指定付き）

### パターン B: セルフレビュー

Claude Code がコードを変更した後、品質チェックのために自発的に実行する。以下の場面で実行する:

- 複数ファイルにまたがる変更を行った後
- リファクタリングを実施した後
- ユーザーの指示が曖昧で意図通りの実装か確信がないとき

---

## ラウンド数

- ユーザーが「Nラウンドでレビューして」のように指定した場合、N ラウンド実行する
- 指定なしの場合は **2 ラウンド**
- 全ラウンド完了後、追加ラウンドを実施するかユーザーに確認する

---

## 実行手順

### ステップ 1: 前提確認

```bash
which codex
```

- 見つかった場合: Agent A, B, C の 3 エージェントで実行
- 見つからない場合: Agent A, B の 2 エージェントのみで実行（ユーザーに codex なしで続行する旨を伝える）

### ステップ 2: ベースブランチの特定

```bash
git rev-parse --verify main 2>/dev/null && echo "BASE=main" || \
  (git rev-parse --verify master 2>/dev/null && echo "BASE=master")
```

どちらも存在しない場合は未コミット変更のみを対象とする。

### ステップ 3: 変更内容の把握

```bash
git diff --name-only <BASE>...HEAD
git diff --name-only HEAD
git ls-files --others --exclude-standard
git log --oneline <BASE>..HEAD
```

変更ファイルが 0 件かつ未コミット変更もない場合は「レビュー対象の変更がありません」と伝えて終了する。

### ステップ 4: 関連ファイルの特定

変更ファイル一覧をもとに、以下の関連ファイルを特定する:

1. **テストファイル**: `*_test.*`, `*.spec.*`, `test_*.*`, `__tests__/` 配下
2. **設定ファイル**: `package.json`, `tsconfig.json`, `Makefile`, `Dockerfile` 等
3. **インポート元**: 変更されたモジュールを import/require しているファイル

```bash
git grep -l "import.*<変更ファイル名>" -- '*.ts' '*.tsx' '*.js' '*.jsx'
git grep -l "from ['\"].*<変更モジュール名>['\"]" -- '*.py'
```

### ステップ 5: 仕様ドキュメントの確認

レビューの精度を上げるため、変更の背景となる仕様ドキュメント（Plan Doc、Specification、Design Doc、RFC 等）の有無を確認する。

#### 5.1 既存コンテキストの確認

現在の会話内で以下のいずれかが既に共有されているか確認する:

- Plan Doc / 設計ドキュメント
- 仕様書 / Specification
- Issue / チケットの説明
- RFC / ADR

共有済みであれば、その内容をステップ 5.2 の「仕様コンテキスト」として使用する。

#### 5.2 未共有の場合: ユーザーに確認

会話内に仕様ドキュメントが見当たらない場合、ユーザーに確認する:

- 「この変更に関する仕様ドキュメント（Plan Doc、Spec 等）はありますか？」
- ある場合: ファイルパスまたは URL を教えてもらい、Read ツールや WebFetch ツールで内容を取得する
- ない場合: コミットメッセージとコード差分のみを根拠にレビューを進める

> **パターン B（セルフレビュー）の場合**: 会話内にタスクの指示内容が既にあるため、それを仕様コンテキストとして使用する。ユーザーへの確認は行わない。

#### 5.3 サブエージェントへの受け渡し

仕様コンテキストが取得できた場合、各サブエージェントのプロンプトに `## 仕様コンテキスト` セクションとして追加する。これにより各エージェントは「仕様に対して実装が正しいか」の観点でもレビューできる。

---

### ステップ 6: レビュー観点の読み込みと分配

`references/` ディレクトリ配下のプロジェクト固有レビュー観点ファイルを読み込む。

```bash
ls ~/.claude/skills/code-review/references/
```

現在の作業ディレクトリのプロジェクト名に一致するファイルがあれば Read ツールで内容を読み込み、以下のマッピングに従って各エージェントのプロンプトに含めるセクションを抽出する。

#### Agent A (Logic) に渡すセクション

- エラーハンドリング
- テスト関連
- DB マイグレーション（up/down 対称性）
- SQLC クエリ
- フォーム実装・Custom Hook
- LIKE パターン
- Valkey 操作

#### Agent B (Security & Design) に渡すセクション

- セキュリティ全般（入力検証・SQL インジェクション・認可・秘匿情報）
- 認証ロジックのバイパス
- マルチテナント分離（tenant_id）
- RLS
- IAM 権限
- URL 構築
- 命名・コメント・用語規約
- ADR
- 破壊的変更
- 既存 UI との整合性
- i18n
- インフラ（既存リソースへの影響）

#### Agent C (Codex) に渡す観点

**全セクション**を `--title` 経由で要約して渡す。

---

### ステップ 7: マルチラウンドレビューの実行

指定ラウンド数（デフォルト 2）だけ以下を繰り返す。

#### 7.1 各エージェントの並列起動

> **重要: 並列実行の指示**
>
> Agent A, B, C の 3 つを **必ず同一メッセージ内で 3 つの Agent ツールコールとして同時に発行すること**。
> 1 つずつ順番に起動してはならない。3 つの Agent ツールコールを 1 回のレスポンスに含めることで並列実行される。

##### Agent A — Logic Reviewer

Agent ツールで起動する。`prompt` に以下の内容を含める:

```

## リポジトリ情報
- パス: <作業ディレクトリの絶対パス>
- ベースブランチ: <BASE>
- 現在のブランチ: <ブランチ名>
- 変更ファイル: <ファイル一覧>
- 関連ファイル: <ステップ4の結果>
- コミット履歴:
<git log --oneline の結果>

1. `git diff <BASE>...HEAD` でコミット済み差分を確認する
2. `git diff HEAD` で未コミット差分を確認する
```

#### 7.2 統合フェーズ

全エージェントの結果が返ってきたら、親 Claude が以下の手順で統合する:

1. **重複排除**: 同一ファイル・同一箇所への同内容の指摘をマージし、発見したエージェントを併記する
2. **重要度の調整**: 複数エージェントが独立して同じ問題を指摘した場合、信頼度が高いため重要度を 1 段階引き上げる（Info → Warning、Warning → Critical）
3. **カテゴリ分類**: 統一カテゴリに整理する
4. **統合レポートの作成**: 次ラウンドの入力として使用する統合レポートを作成する

統合レポートの形式:

```
## 統合レポート（ラウンド N）

### Critical
1. ファイル:行 — 指摘内容 [発見: Agent A, B]
2. ...

### Warning
1. ファイル:行 — 指摘内容 [発見: Agent C]
2. ...

### Info
1. ファイル:行 — 指摘内容 [発見: Agent B]
2. ...
```

#### 7.3 次ラウンドへの遷移

最終ラウンドでなければ、統合レポートを各エージェントのプロンプトに追加してステップ 7.1 に戻る。

---

### ステップ 8: 最終レポートの出力

全ラウンド完了後、以下の形式で最終レポートを出力する:

```
## コードレビュー結果（全 N ラウンド完了）

### Critical（要対応）
- [ ] ファイル:行 — 指摘内容 [発見: Agent A, B]
- [ ] ...

### Warning（検討推奨）
- [ ] ファイル:行 — 指摘内容 [発見: Agent C]
- [ ] ...

### Info（参考）
- ファイル:行 — 指摘内容 [発見: Agent B]
- ...

### 統計
- Critical: N 件 / Warning: N 件 / Info: N 件
- ラウンド間の推移: ラウンド1 → ... → 最終ラウンド
```

### ステップ 9: 追加ラウンドの確認

最終レポート出力後、ユーザーに「追加ラウンドを実施しますか？」と確認する。

- 「はい」「追加で」等の場合: ステップ 7 に戻り 1 ラウンド追加実行する（再度確認も行う）
- 「いいえ」の場合: レビュー終了

---

## パターン B: セルフレビュー

セルフレビューでは **1 ラウンド** のマルチエージェントレビューを実行し、その後修正ループに入る。

1. Agent A, B, C を並列起動（1 ラウンドのみ）
2. 統合レポートを作成
3. 各指摘を分類する:
   - **【修正する】**: バグ、論理エラー、要件の未充足、セキュリティ問題 → 修正を実施
   - **【対応不要】**: スコープ外、既存コードの規約に沿っている、トレードオフを考慮した上での設計判断
4. 修正すべき指摘がある場合:
   a. 修正を実施する
   b. Agent A, B, C を再度並列起動して再レビュー
   c. 3 に戻る
5. 全指摘が解消 or 対応不要になったらループ終了
6. ユーザーに以下を報告する:
   - 修正した内容の一覧（何をどう直したか）
   - 対応不要と判断した指摘の一覧と理由
   - 最終レビュー結果（問題なし or 残課題あり）

---

## エラーハンドリング

| 状況 | 対応 |
|------|------|
| `codex` が見つからない | Agent A, B のみで実行（Agent C をスキップ）。ユーザーに通知する |
| ベースブランチが見つからない | 未コミット変更のみを対象にレビュー |
| 変更が 0 件 | 「レビュー対象の変更がありません」と伝えて終了 |
| サブエージェントがエラー | 残りのエージェント結果で統合を続行。エラーを報告する |
| codex がタイムアウト | Agent A, B の結果で統合し、codex スキップを報告 |
